1 · rivers_timer

小型 standalone SysTick timer module。默认自动配置 SysTick，SysTick 中断只推进系统时间；软件定时器到期判断和回调执行由主循环中的 poll 完成。

2 · rivers_queue

standalone mem / queue module。包含静态内存池分配器和固定单元环形队列；mem 动态效果和 queue 环形队列效果在本页中分开展示。

0.1 · 总体架构：轻量 standalone 基础模块

定位边界：这四个文件整体上不是完整操作系统，而是两个独立的 standalone 基础模块：queue + mem 和 timer。它们的设计目标是给裸机 / CubeMX 工程提供轻量的“中断与主循环协作”能力。

queue + mem

rivers_mem 位于 rivers_queue.c 内部，是静态内存池分配器；既可以使用模块内置的 s_default_heap，也可以使用用户传入的外部 buffer。队列控制块和队列数据区都从这里分配。

rivers_queue 是固定长度、固定 item 大小的环形队列。创建后容量和单元大小不变，send/recv 只做立即尝试，不等待、不阻塞、不覆盖旧数据。

裸机前后台协作
中断只做最小工作

timer

rivers_timer_init() 直接配置 Cortex-M SysTick 寄存器，SysTick_Handler() 中调用 rivers_timer_tick_handler() 推进 uptime。

软件定时器使用静态数组 s_sw_timers[]，不动态分配。定时器回调不在 SysTick 中断中执行，而是在主循环调用 rivers_timer_poll() 时执行。

rivers_queue.c ├── rivers_mem：静态内存池分配器 └── rivers_queue：固定 item 环形队列
rivers_timer.c ├── SysTick 时间基准 └── s_sw_timers[] 软件定时器数组

不依赖

  • HAL
  • CMSIS
  • malloc/free

不提供

  • 任务阻塞
  • 任务调度
  • timeout 等待
  • 任务调度语义
这套代码面向裸机前后台协作，适合做小型 OSAL 基础模块。
边界只需要记住一点：这些模块只负责“时间、内存、消息缓冲”这些基础能力，不负责决定哪个任务运行，也不负责唤醒或切换执行上下文。

0.2 · 典型使用方式

int main(void)
{
    HAL_Init();
    SystemClock_Config();

    rivers_mem_init(NULL, 0);
    rivers_timer_init();

    q = rivers_queue_create(128, sizeof(my_msg_t));

    timer_id = rivers_sw_timer_create(1000, true, cb, NULL);
    rivers_sw_timer_start(timer_id);

    while (1) {
        rivers_timer_poll();

        while (rivers_queue_recv(q, &msg) == RIVERS_QUEUE_OK) {
            // 主循环处理消息
        }
    }
}

void SysTick_Handler(void)
{
    rivers_timer_tick_handler();
}

void Some_IRQHandler(void)
{
    rivers_queue_send_from_isr(q, &msg);
}

中断侧

中断只做快速入队或更新时间。SysTick_Handler() 只推进系统时间，外设 ISR 可以把消息放进队列。

主循环侧

rivers_timer_poll() 在主循环里触发软件定时器回调，队列作为 ISR 与主循环之间的消息缓冲，真正业务处理放在主循环。

裸机前后台协作流程

中断只做最小工作，主循环承担真正业务处理。

0.3 · 关键边界

使用时必须知道的边界

边界点含义
裸机主循环模型适合前后台协作：中断快速记录事件，主循环集中处理业务。
队列不阻塞队列不会等待。满了直接返回 RIVERS_QUEUE_ERR_FULL，空了直接返回 RIVERS_QUEUE_ERR_EMPTY。
软件定时器依赖 poll软件定时器精度取决于 rivers_timer_poll() 被调用得多频繁。
SysTick 只推进时间SysTick 中断只更新 uptime，不执行用户回调，也不承载业务逻辑。
from_isr 是立即尝试send_from_isr() 语义是“可以在 ISR 中立即尝试发送”，不包含任务唤醒或调度语义。
rivers_mem 不是 malloc它是在静态内存池上的可控动态分配，不使用系统堆。
queue 创建后固定length 和 item_size 固定不变；满时不覆盖旧数据。
一句话概括：这四个文件把静态内存管理、消息队列、系统时间、软件定时器拆成可独立移植的小模块；核心思想是用临界区保护共享状态，用主循环承担业务处理，用中断只做最小工作。

rivers_timer

rivers_timer 的核心定位是：用 Cortex-M SysTick 建立系统时间基准，提供 uptime、delay 和 ms 级软件定时器。它不使用 malloc/free，软件定时器使用静态数组，回调不在 SysTick 中断中执行。

1.1 · 支持功能

✓自动配置 Cortex-M SysTick，初始化时计算 reload，写入 LOAD / VAL / CTRL。
✓使用 SysTick 作为系统时基，固定打开 ENABLE、TICKINT、CLKSOURCE。
✓需要在 SysTick_Handler() 中调用 rivers_timer_tick_handler() 推进系统时间。
✓维护 32 位 us/ms uptime，并允许自然回绕。
✓维护 64 位 us/ms uptime，给软件定时器内部绝对到期时间比较使用。
✓us 转 ms 时保留 s_ms_remainder_us，避免长期丢弃不足 1ms 的误差。
✓提供 rivers_timer_get_tick()、rivers_timer_get_uptime_ms()、rivers_timer_get_uptime_us()。
✓提供 rivers_timer_delay_ms() 和 rivers_timer_delay_us() 忙等延时。
✓提供 ms 级软件定时器，支持单次和周期模式。
✓软件定时器来自静态数组 s_sw_timers[]，容量由 RIVERS_TIMER_MAX_SW_TIMERS 控制，默认 16。
✓不使用 malloc/free，不依赖 HAL、CMSIS 或原 rivers_osal。
✓软件定时器回调在 rivers_timer_poll() 中执行，不在 SysTick 中断中执行。

1.2 · 接口说明

接口作用典型调用位置阻塞 / 回调注意事项
rivers_timer_init()清零 uptime、清空软件定时器静态表、配置 SysTick。系统初始化。不阻塞，不执行用户回调。会配置 SysTick；若工程已有 SysTick 管理，需要统一。
rivers_timer_tick_handler()推进 us/ms uptime。SysTick_Handler()。不执行用户回调。只更新时间，不检测软件定时器是否到期。
rivers_timer_get_tick()返回 32 位 ms tick。普通/中断上下文。不阻塞，不回调。语义等同于 get_uptime_ms。
rivers_timer_get_uptime_ms()读取已累计的 32 位 ms uptime。时间戳、超时判断。不阻塞，不回调。不读取 SysTick VAL，只读累计变量。
rivers_timer_get_uptime_us()读取整 tick us，并结合 VAL 估算 tick 内 extra_us。短时间测量、us delay。不阻塞，不回调。返回 now_us + extra_us。
rivers_timer_delay_ms()基于 tick 差值忙等 ms。短暂等待。阻塞 CPU，不回调。不调用 HAL_Delay，不执行软件定时器回调。
rivers_timer_delay_us()基于 uptime_us 差值忙等 us。短暂硬件等待。阻塞 CPU，不回调。真正 us 周期任务应使用硬件定时器。
rivers_timer_poll()扫描软件定时器静态数组，处理到期定时器。while(1) 或任务态。可能执行用户回调。软件定时器到期判断发生在这里。
rivers_sw_timer_create()创建单次或周期软件定时器。初始化或业务创建阶段。不阻塞，不回调。回调函数不能为 NULL。
rivers_sw_timer_start()设置 expiry_ms = now_ms + period_ms 并置 active。启动定时器。不阻塞，不回调。timer_id 必须有效且已分配。
rivers_sw_timer_stop()停止定时器，保留 allocated。暂停定时器。不阻塞，不回调。可再次 start。
rivers_sw_timer_delete()删除定时器，清空槽位。不再需要定时器。不阻塞，不回调。delete 后 timer_id 失效。
rivers_sw_timer_set_period()修改周期/超时时长。动态调整定时器。不阻塞，不回调。active 时会重算下一次 expiry_ms。
rivers_sw_timer_set_remaining()修改 active 定时器的剩余时间。重设倒计时。不阻塞，不回调。未 active 返回 NOT_READY。

1.3 · 软件思维

中断只做账本更新

SysTick_Handler() 只转调 rivers_timer_tick_handler()，让 uptime 前进；业务和回调不放进中断。

回调由主循环 poll

rivers_timer_poll() 扫描 s_sw_timers[]，到期后先更新状态，再在主循环上下文执行 callback。

时间读取保护一致性

读取 64 位时间或组合“整 tick + 子 tick”时进入临界区，避免和 SysTick 中断更新账本交错。

本节只讲设计原因。具体的 reload、period_us、余数累计、VAL 子节拍和 COUNTFLAG 计算集中放在 1.6。
软件定时器控制块包含 allocated、active、periodic、expiry_ms、period_ms、cb、arg。周期定时器到期后推进 expiry_ms，单次定时器到期后 active=false。

1.6 · 源码与计算细节

这一节按源码执行顺序串起来：先配置 SysTick，再把硬件计数周期换算成微秒，tick 中断只累计时间，读取 us 时再补当前 tick 内的子节拍。

0. 四个 uptime 变量为什么同时存在

static volatile uint32_t s_uptime_us32 = 0U;
static volatile uint32_t s_uptime_ms32 = 0U;
static volatile uint64_t s_uptime_us64 = 0U;
static volatile uint64_t s_uptime_ms64 = 0U;
变量名单位位宽主要用途回绕特点
s_uptime_us32us32 bit短时间测量、delay_us、快速读取微秒时间。约 71 分钟自然回绕；用差值判断时可以正常工作。
s_uptime_ms32ms32 bit普通 tick、uptime_ms、delay_ms。约 49.7 天自然回绕；不是 bug，是 32 位计数器的正常行为。
s_uptime_us64us64 bit保存更完整的微秒运行时间，方便后续扩展长时间微秒统计。实际使用中很难遇到回绕。
s_uptime_ms64ms64 bit软件定时器内部用它做 expiry_ms 绝对时间比较，减少长期运行回绕问题。适合长期运行的到期时间账本。

1. SysTick 初始化：reload / LOAD / period_ticks

reload = RIVERS_TIMER_SYSTICK_CLOCK_HZ / RIVERS_TIMER_TICK_RATE_HZ;
LOAD = reload - 1;
period_ticks = LOAD + 1;
STM32F407, 168MHz, 1000Hz tick: reload = 168000000 / 1000 = 168000 LOAD = 168000 - 1 = 167999 period_ticks = 167999 + 1 = 168000

SysTick 是递减计数器，LOAD 写 reload - 1。从 167999 递减到 0，一共经历 168000 个硬件计数。

2. 周期换算：整数四舍五入

period_us =
    (period_ticks * 1000000 + counter_hz / 2) / counter_hz;

period_ticks * 1000000 / counter_hz 是把硬件计数周期换算成微秒。+ counter_hz / 2 是整数四舍五入，避免永远向下截断。

默认: period_us = (168000 * 1000000 + 168000000 / 2) / 168000000 = 1000us

3. tick 累计：us 直接加，ms 只加够 1000us 的部分

s_uptime_us32 += delta_us;
s_uptime_us64 += (uint64_t)delta_us;

total_us = s_ms_remainder_us + delta_us;
elapsed_ms = total_us / 1000U;
s_uptime_ms32 += elapsed_ms;
s_uptime_ms64 += (uint64_t)elapsed_ms;
s_ms_remainder_us = total_us % 1000U;
delta_us = 600: 第 1 次: total=600, ms += 0, remainder=600 第 2 次: total=1200, ms += 1, remainder=200 第 3 次: total=800, ms += 0, remainder=800 第 4 次: total=1400, ms += 1, remainder=400

4. get_uptime_us = 整 tick + 当前 tick 内子节拍

now_us = s_uptime_us32;
extra_us = rivers_timer_get_subtick_us_locked();
return now_us + extra_us;
LOAD = 167999 VAL 从 167999 递减到 0 elapsed_ticks = LOAD - VAL extra_us = elapsed_ticks * 1000000 / counter_hz

s_uptime_us32 是已经由 SysTick 中断累计的整 tick 时间，extra_us 是当前 tick 周期内已经过去的时间。COUNTFLAG 用来判断读取期间是否刚好跨过一次 SysTick 回卷，源码通过 current_before / COUNTFLAG / current_after 缩小边界误差。

边界：这个子 tick 估算只能补当前 tick 内的偏移，或者约一个 tick 范围内的短暂中断延迟；不能补偿长时间关中断导致错过多个 SysTick 周期。

2.3 · 软件思维

统一内存来源

队列控制块和 storage 都来自 rivers_mem，不使用系统堆。

不透明句柄 + 活动链表

外部只拿 rivers_queue_t *；s_queue_list 只记录当前仍有效的 queue 句柄。

固定单元拷贝队列

创建后 length/item_size 固定；send/recv 复制完整 item，不等待、不覆盖旧数据。

本节只讲设计取舍。块头、地址偏移、对齐、两次分配、head/tail/count 的计算放在后面的源码解析小节。

2.4 · rivers_mem：静态内存池上的动态分配

rivers_mem 属于 rivers_queue.c 内部基础能力：总 RAM 固定，但运行时可以在这块 RAM 里按需 alloc/free 不同大小的块。

A. 内存来源

static rivers_default_heap_t s_default_heap;
uint8_t bytes[RIVERS_QUEUE_HEAP_SIZE];

默认使用内部静态数组，也可以由 rivers_mem_init(buffer, size) 传入外部 buffer。它不是系统 malloc/free，而是在固定 RAM 中维护 block 链表。

A1. rivers_mem 的模块级状态变量

static rivers_default_heap_t s_default_heap;
static uint8_t *s_heap_start = NULL;
static uint32_t s_heap_size = 0U;
static rivers_mem_block_t *s_heap_head = NULL;
static bool s_mem_ready = false;
static uint32_t s_min_free_size = 0U;
static rivers_queue_t *s_queue_list = NULL;
变量含义
s_default_heap内部默认静态堆，rivers_mem_init(NULL, 0) 时使用。
s_heap_start当前内存池对齐后的真实起点。
s_heap_size当前内存池对齐后的真实可用总大小。
s_heap_head内存块链表头，初始化后指向第一个大的 free block。
s_mem_ready内存池是否初始化成功；alloc/free/get_free_size 等接口会根据它判断是否可用。
s_min_free_size自最近一次 rivers_mem_init() 以来的历史最低剩余空间，用来观察内存压力峰值。
s_queue_list当前有效 queue 句柄链表，用于判断 queue 是否仍然 active，降低 delete 后继续使用野指针的问题。

B. 块头 rivers_mem_block_t 与 header_size

typedef struct rivers_mem_block {
    uint32_t size;
    bool free;
    struct rivers_mem_block *next;
} rivers_mem_block_t;

header_size = rivers_align_up_u32(sizeof(rivers_mem_block_t));

size 是当前块总大小，包含 header 和 payload；free 表示是否空闲；next 指向下一个块。常见 32 位 MCU 上 sizeof(rivers_mem_block_t) 可能是 12 字节，但源码通过 rivers_mem_header_size() 得到对齐后的 header_size。默认 8 字节对齐时，header 常见会按 16 字节计算。

C. 对齐计算：向上、向下和 2 的幂理解

rem = value % align;
align_up = (rem == 0U) ? value : (value + (align - rem));

align_down = value - (value % align);
当 align 是 2 的幂: x & (align - 1) 得到余数 x & ~(align - 1) 向下对齐 (x + align - 1) & ~(align - 1) 向上对齐 8 字节对齐: 13 & 7 = 5 13 & ~7 = 8 (13 + 7) & ~7 = 16

D. mem_init：修正 buffer 起点和真实可用大小

raw_start = (uintptr_t)buffer;
aligned_start = rivers_align_up_ptr(raw_start);
adjust = (uint32_t)(aligned_start - raw_start);
aligned_size = rivers_align_down_u32(size - adjust);

raw_start 是用户 buffer 原始起点；aligned_start 是向上对齐后的堆起点；adjust 是为对齐浪费掉的前部字节；aligned_size 是减去 adjust 后再向下对齐得到的真实可用大小。

s_heap_start = aligned_start;
s_heap_size = aligned_size;
s_heap_head = s_heap_start;
s_heap_head->size = s_heap_size;
s_heap_head->free = true;
s_heap_head->next = NULL;

D1. 为什么地址要先转成 uintptr_t

raw_start = (uintptr_t)buffer;
aligned_start = rivers_align_up_ptr(raw_start);
adjust = (uint32_t)(aligned_start - raw_start);

rivers_align_up_u32() 对齐的是数值大小，例如 size、payload_size、header_size；rivers_align_up_ptr() 对齐的是地址，例如用户传入的 buffer 起始地址。

uintptr_t 是一种能安全保存指针地址的无符号整数类型。把指针转成 uintptr_t 后，才能对地址做取余、加减、对齐计算。不要用 uint32_t 直接保存指针地址，否则在 64 位平台上可能截断地址。

raw_start = 0x20000003 8 字节对齐后: aligned_start = 0x20000008 adjust = 5

adjust 就是为了让 heap 起点满足对齐要求而浪费掉的前部字节。

E. mem_alloc：用户 size 不是实际申请总量

header_size = rivers_mem_header_size();
payload_size = rivers_align_up_u32(size);
request_size = rivers_align_up_u32(header_size + payload_size);
假设: RIVERS_QUEUE_MEM_ALIGN_SIZE = 8 sizeof(rivers_mem_block_t) = 12 header_size = align_up(12) = 16 用户申请 size = 100 payload_size = align_up(100) = 104 request_size = align_up(16 + 104) = 120
block 起点0x20000000
payload 起点0x20000010
next block0x20000078

return (uint8_t *)block + header_size; 的意思是：block 指向 header，用户不能写 header，所以返回 header 后面的 payload 起点。

F. split / free / coalesce

remain = block->size - request_size;
next = (rivers_mem_block_t *)((uint8_t *)block + request_size);

request_size 是本次分配块的总大小，所以 block + request_size 正好是剩余空闲区域的新 block 起点。

block = (rivers_mem_block_t *)(p - header_size);

释放时用户传入的是 payload 指针，payload = block + header_size，所以 ptr - header_size 就能反推出 block header。

block_end = (uint8_t *)block + block->size;
if (block_end != (uint8_t *)block->next) break;

只有物理地址连续的两个空闲块才能合并。链表上 next 相邻不代表内存地址一定连续。

F1. free 不是直接相信用户指针

  1. ptr == NULL 时直接返回，free(NULL) 是安全空操作。
  2. 确保内存池已经初始化，s_mem_ready == true。
  3. 检查 ptr 是否落在当前 heap 范围内。
  4. 检查 ptr 是否至少位于第一个 payload 之后，避免指到 header 区域之前。
  5. 用 ptr - header_size 反推出 block header。
  6. 遍历 block 链表，确认这个 block 真的存在于当前内存池链表中。
  7. 如果 block 不存在，说明 ptr 不是 rivers_mem_alloc() 返回的有效指针，安全返回并通过 debug hook 上报。
  8. 如果当前 block 已经是 free，避免重复释放。
  9. 标记 current->free = true。
  10. 尝试和后面的连续空闲块合并。
不能随便 free 一个地址。用户传入的是 payload 指针，内存管理器必须先反推出 header，再确认这个 header 确实属于当前块链表。

G. 一个队列到底会占多少内存

queue_cost ≈ align_up(length * item_size) + align_up(sizeof(rivers_queue_t)) + 2 * header_size

因为 queue_create 有两次 rivers_mem_alloc()：一次申请 storage 数据区，一次申请 rivers_queue_t 控制块，所以会有两个 block header。

示例storage payloadstorage blockqueue control block总计估算
RX byte queue
length=128, item_size=1		align_up(128 * 1) = 12816 + 128 = 14416 + align_up(sizeof(rivers_queue_t)) ≈ 48约 192 字节
TX message queue
length=8, item_size=66		align_up(8 * 66) = align_up(528) = 52816 + 528 = 544约 48 字节约 592 字节
sizeof(rivers_queue_t) 受平台和编译器对齐影响，示例按默认 8 字节对齐、header_size = 16 的常见 32 位场景估算。

H. 和 FreeRTOS heap_4 的相似点

相似点

  • 都在一块静态 RAM 上做动态分配。
  • 每个分配块前面都有 block header。
  • alloc 时会从空闲空间中切出一块。
  • free 后会尝试合并相邻空闲块，减少碎片。

区别

  • 当前 rivers_mem 更偏学习型和轻量 standalone 实现，维护的是完整 block 链表。
  • heap_4 更成熟，空闲块链表管理和边界处理更完整。

// 创建字节队列
rivers_mem_init(NULL, 0);

    }

2.7 · rivers_queue：固定单元环形消息队列

这一节只讲 queue 本身：控制块字段、创建时两次分配、send/recv 的字节偏移，以及 head/tail/count 如何描述队列状态。

struct rivers_queue {
    uint8_t *storage;
    uint32_t head;
    uint32_t tail;
    uint32_t length;
    uint32_t item_size;
    uint32_t count;
    struct rivers_queue *next;
};
字段含义
storage连续数据区，真正保存 item 内容副本。
head下一个要读的槽位编号。
tail下一个要写的槽位编号。
length槽位数量。
item_size每个槽位的字节数。
count当前已有元素数量，用来区分空和满。
next活动队列链表指针，用于检查句柄是否仍有效。

1. queue_create：storage 和控制块是两次分配

total_size = length * item_size;
storage = rivers_mem_alloc(total_size);
q = rivers_mem_alloc(sizeof(rivers_queue_t));

一个 queue 由两块内存组成：storage 存队列元素，rivers_queue_t 控制块保存 head/tail/count 等状态。创建成功后挂入 s_queue_list。

2. send：tail 是槽位编号，乘 item_size 才是字节偏移

memcpy(q->storage + (q->tail * q->item_size), item, q->item_size);
q->tail = (q->tail + 1U) % q->length;
q->count++;
length = 4, item_size = 8, tail = 2 写入地址 = storage + 2 * 8 = storage + 16

% length 用来在到达末尾后绕回 0；count++ 表示队列里多了一个元素。

3. recv：head 转字节偏移后读取

memcpy(item, q->storage + (q->head * q->item_size), q->item_size);
q->head = (q->head + 1U) % q->length;
q->count--;

head 是下一个要读的槽位编号。读取后 head 后移一个槽位，count-- 表示队列里少了一个元素。

如果只看 head == tail，无法判断到底是空队列还是满队列；本实现用 count == 0 判断空，用 count >= length 判断满。

4. 活动队列链表不是调度链表

s_queue_list
rivers_queue_link_locked()
rivers_queue_unlink_locked()
rivers_queue_is_active_locked()

这个链表只记录当前仍有效的 queue 句柄。send/recv/delete 时检查 q 是否 active，可以降低 delete 后继续使用野指针的风险。

5. delete 如何安全摘除队列

active = rivers_queue_unlink_locked(q);
storage = active ? q->storage : NULL;

s_queue_list 不是调度链表，而是当前有效 queue 句柄链表。delete 时先进入临界区，然后通过 rivers_queue_unlink_locked(q) 尝试把 q 从活动链表中摘除。

如果 q 在链表里，说明它还是 active；如果不在链表里，说明它可能已经被 delete，或者不是合法 queue 句柄。

if (prev == NULL) {
    s_queue_list = cur->next;
} else {
    prev->next = cur->next;
}

prev == NULL 表示要删除的节点刚好是链表头，需要更新 s_queue_list = cur->next。prev != NULL 表示节点在中间或尾部，只需要让前一个节点跳过当前节点。

storage = active ? q->storage : NULL; 是三目表达式赋值。只有 active 为 true 时才读取并释放 q->storage；如果 active 为 false，就不继续相信这个句柄，避免对无效 q 继续解引用。
